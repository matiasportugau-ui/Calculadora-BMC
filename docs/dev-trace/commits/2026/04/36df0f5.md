## Descripción
Este cambio registra el commit `feat(deploy): graceful Secret Manager migration for high-sensitivity keys` dentro del sistema de trazabilidad del proyecto. Se modificaron 3 archivos: docs/procedimientos/SECRETS-MIGRATION.md, run_ml_cloud_run_setup.sh, scripts/provision-secrets.sh.

Contexto del commit:
Until now run_ml_cloud_run_setup.sh shipped ANTHROPIC/OPENAI/GEMINI/GROK
API keys, ML_CLIENT_SECRET, TOKEN_ENCRYPTION_KEY, API_AUTH_TOKEN,
WEBHOOK_VERIFY_TOKEN and the WhatsApp tokens as plain Cloud Run env
vars — visible to anyone with run.services.get on the project.

Now:
- run_ml_cloud_run_setup.sh introduces add_sensitive() — auto-detects
  whether each high-sensitivity key has a Secret Manager secret. If yes,
  mounts via --update-secrets KEY=KEY:latest; if not, falls back to env
  var with a warning. The current deploy keeps working without any GSM
  setup; each key migrates independently as you provision it.
- scripts/provision-secrets.sh (new): idempotent helper that reads .env,
  creates/updates the GSM secrets, and grants secretmanager.secretAccessor
  to the Cloud Run runtime SA.
- docs/procedimientos/SECRETS-MIGRATION.md: runbook covering scope,
  migration steps, rotation, and rollback.

Out of scope: WHATSAPP_APP_SECRET (already tracked separately in
WHATSAPP-HMAC-GAP.md) and the panelin-service-account JSON (already on
GSM via the CI workflow).
