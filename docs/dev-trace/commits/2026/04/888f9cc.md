Contexto del commit:
Three v1.3 Fase 0 audit docs plus the related script tweaks:

- docs/procedimientos/OAUTH-CHECKLIST.md (Fase 0 #1) — single source of
  truth for every OAuth callback and webhook URL the API consumes.
  Inventories ML, Shopify, WhatsApp, plus the absolute URLs to register
  in each external console. Flags 4 security gaps (ML signature not
  validated, WA HMAC conditional, WEBHOOK_VERIFY_TOKEN empty, OAuth
  state in-memory only).

- docs/team/AUDIT-AREA-4.md (Fase 0 #2) — data-flow inventory of the KB
  (training-kb.json + GCS), conversation JSONL, response feedback,
  prompt sections, and 4 admin route surfaces. Includes the proposed
  Supabase + pgvector schema for Fase 2 (kb_entries, kb_entry_embeddings,
  conversations, conversation_turns, conversation_actions,
  response_feedback, kb_documents, training_sessions).

- docs/adr/0001-source-of-truth-post-supabase.md (Fase 0 #4) —
  per-entity SoT decisions for the post-Supabase world. Status: proposed.
  Locks dual-write windows + cutover criteria so each Fase 1 PR doesn't
  re-litigate the same questions.

- run_ml_cloud_run_setup.sh + scripts/provision-secrets.sh (Fase 0 #3
  tail) — added WHATSAPP_APP_SECRET, SHOPIFY_CLIENT_SECRET, and
  SHOPIFY_WEBHOOK_SECRET to the high-sensitivity provisioning + sync
  arrays so they get GSM-backed on the next run.

Phase A (Cloud Run secrets re-deploy with --update-secrets mounts) is
still pending explicit user authorization.
