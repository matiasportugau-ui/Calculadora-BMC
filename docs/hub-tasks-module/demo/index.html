📊 Sync Workflow

Scenario: Scheduled Sync Cycle

Cloud Scheduler triggers POST /sync/google-tasks/pull every 60 seconds. System fetches changes from Google Tasks API and syncs into HUB database.

T+0s Cloud Scheduler fires → POST /sync/google-tasks/pull with HMAC-SHA256 signature
T+1s Load active oauth_tokens (revoked_at IS NULL)
T+2s For each user: decrypt access_token (pgp_sym_decrypt)
T+3s Query Google Tasks API (GET /tasks/v1/users/@me/lists) with updatedMin
T+4s Upsert into tasks.task_lists + tasks.tasks (ON CONFLICT DO UPDATE)
T+5s Detect conflicts: is_deleted=true but Google version active
T+6s Insert into tasks.sync_conflicts, notify UI (TanStack Query invalidation)
T+7s Log cycle: tasks.sync_log(event_type='sync_completed', cycle_id)

⚔️ Conflict Resolution

Scenario: Soft-Delete Mismatch

User deletes task in HUB (soft-delete: is_deleted=true), but task was updated in Google Tasks before sync caught it. System creates conflict record with 7-day TTL.

⏱️ Rate Limiting (1 req/min)

Scenario: Manual Sync Trigger

POST /sync/google-tasks/pull is rate-limited to 1 request per minute per user. Exceeding the limit returns 429 Too Many Requests with retry-after header.

🏗️ Architecture Notes

  • Bidirectional Sync: Cloud Scheduler polls every 60s (no native webhooks in Tasks API v1)
  • Pagination: nextPageToken for large task lists (>100 items per list)
  • Encryption: pgp_sym_encrypt for OAuth tokens in transit and at-rest
  • Conflict TTL: 7-day expiry on sync_conflicts; operator must resolve before expiry
  • RLS Enforcement: Service-role queries (Cloud Scheduler, API) bypass RLS; user queries enforce user_id = current_user_id
  • TanStack Query: staleTime=60s, gcTime=5min; invalidation on mutation success triggers fresh fetch
  • Offline Resilience: IndexedDB queue persists pending mutations across browser restarts
  • Token Refresh: 401 from Google Tasks API triggers refresh_token flow; revokes token if refresh fails
