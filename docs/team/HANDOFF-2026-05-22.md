## Three corrections that landed during the session

1. **Subtask 1 turned out to be a 2-bug compound, not a 1-bug fix.** When I started, I assumed matias's refresh_token was stored correctly and I just needed to add refresh-on-401 logic. The Phase 0 assumption check decrypted his refresh_token_encrypted column and got an EMPTY STRING — Google had only issued a refresh_token on the first consent, and subsequent reconnects (during the OAuth debugging in the prior session) overwrote it with NULL via the upsert. Fixed by adding COALESCE to the ON CONFLICT UPDATE so reconnect-without-new-refresh-token preserves the existing one. **Side effect:** matias still has no refresh_token stored even after this fix lands — he needs to revoke + reconnect once to capture a fresh refresh_token under the new code. That's the verification step that didn't run tonight because there was no point doing it before the code was in production.
2. **Subtask 2's edit had a subtle ordering bug to solve.** The upsert that bumps `synced_at = now()` was returning the new value, not the old. So my first cut would have always used `priorSyncedAt = now()`, never EPOCH_ISO. Fixed by adding a SELECT before the upsert to capture the prior value (NULL on first sync, real timestamp on incremental).
3. **Subtask 5 advisor call was over-validation.** I called `advisor()` per the goal prompt's "advisor before one-way door" rule. Advisor pushed back: schema flip from RLS=false to RLS=true is reversible (`DISABLE ROW LEVEL SECURITY`), policies are additive, postgres role has BYPASSRLS so runtime is unaffected — there's no judgment call for an advisor to make. Applied directly afterward. **Lesson logged for the procedure**: the gate should be "advisor for irreversible decisions only" (column drops, data migrations, encryption key rotations). RLS flips don't qualify.
